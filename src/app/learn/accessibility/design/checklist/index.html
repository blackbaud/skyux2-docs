Designers should follow these guidelines when designing for SKY UX. Text contrast ratio

  • If text is less than 18 points (or less than 14 points for bold text), then the contrast ratio of background to text should be greater than 4.5:1. See 1.4.3..

  • If text is greater than 18 points (or greater than 14 points for bold text), then the contrast ratio of background to text should be greater than 3:1. see 1.4.3.

Links

  • If links within text (such as paragraphs) are not underlined, then the link color to text color contrast should be greater than 3:1, and links should have additional visual indication when hovered or focused. See 1.4.1.

Images

  • Images and nontext content such as icons should have text alternatives defined. (If images are decorative, indicate alt attribute would be blank.) See 1.1.1.

  • If images and nontext content such as icons invoke functions, then they should have text alternatives that communicate the function. See 1.1.1.

Color

  • If color is used to convey information, it should not be the only means of communicating the information. See 1.4.1.

Motion and animations

  • Flashing motions and animations, should not flash more than 3 times in any 1 second period. See 2.3.1.

  • For motion, blinking, or scrolling that starts automatically and lasts more than 5 seconds, a mechanism should allow users to pause, stop, or hide the motion unless it is essential. See 2.2.2.

  • For auto-updating content that automatically starts, a mechanism should allow users to pause, stop, or hide the motion unless it is essential. See 2.2.2.

Forms and input elements

  • All input elements should have text labels onscreen or specified off-screen. See 3.3.0.

  • If form input elements and input formats are required, the requirement should be communicated visually and textually. See 3.3.2, and see 1.4.1.

Error states and messages

  • When users are notified of errors and given instructions on how to fix errors, a method should ideally exist to navigate directly to errors from error messages. See 3.3.1, and see 3.3.3.

Interactions

  • Interactions that are mouse or touch-only, such as drag and drop actions, should inclue keyboard operation alternatives in the design proposal. See 2.1.1.

  • When focus is placed on interactive elements such as links, buttons, and input elements, no unexpected or non-browser-default behaviors should be specified. See 3.2.1.

  • When focus is placed on interactive elements such as links, buttons, and input elements, additional focus state visual indicators should specified as necessary. See 2.4.7.

  • During text entry or input selection for interactive elements such as links, buttons, and input elements, no unexpected behavior should be invoked. If unexpected behavior must be invoked, a warning should be communicated to users ahead of time. See 3.2.2.

Page titles

  • Titles should be unique and should describe the contents and purpose of the page.

Keyboard support

  • Interactions for keyboard and focus indication should be defined following WAI-ARIA Authoring Practices for non-standard HTML component patterns. See 2.1.1, and see 2.4.7.

Headings

  • Heading levels should be specified and should follow hierarchical order from H1 to H6. See 1.3.1.
