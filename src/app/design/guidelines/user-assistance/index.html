When users access functionality for the first time, they may require guidance. To determine how much guidance to provide and how to present it, you must account for the complexity of the task, the complexity of the user interface, and the expected knowledge of users. User assistance design patterns provide guidance on how to present help content in SKY UX SPAs.

Usage

SKY UX design patterns include four primary techniques to display help content: empty-state help, inline help, preloaded Help Widget, and popovers. To determine the appropriate technique for user assistance, consider the complexity of the task and the expected knowledge of users. The chart below provides a matrix to determine which type of user assistance to use in different scenarios. Do not overuse help content because it can overwhelm or inhibit users. For example, only display the preloaded Help Widget and popovers when user research supports the need for it. When in doubt, err on the side of empty-state help and inline help.

Empty-state help
Usage

Empty-state help provides user assistance when content containers are not populated. The content can include a title, body copy, and buttons to take actions or access additional resources. The help content is not available when content containers are populated.

Layout

Center empty-state help in its content container.

Inline help

Page-level and modal-level help

When users perform a complex task, they may require comprehensive documentation for the entire task instead of inline help for individual parts of the task. Users should invoke help at the page or modal level, depending on the context.

In Blackbaud products, users can invoke the help service from the omnibar or a modal header bar.

Invoked inline help

Usage

Invoked inline help gives users access to help content when needed while avoiding most of the visual noise that comes with persistent inline help text. It is appropriate to use invoked inline help in cases where users may need initial assistance but:

  • The task is performed frequently.
  • Ramifications of mistakes are small.

In general, use a help button to invoke the help content. For more complex help content, you can also use a popover or the help panel.

Layout

Place the help button to the right of the label for the element that the help content supports.

Persistent inline help

Usage

Sometimes a small amount of persistent inline explanatory text can help users understand tasks. It is appropriate to use persistent inline help in cases where:

  • The task is complex and is performed infrequently.
  • User actions have far-reaching consequences that are not obvious.

Use the "sky-deemphasized" class for persistent inline help, and limit the content to one or two sentences to minimize the visual weight of the text.

Layout

The placement of persistent inline help text depends on the aspect of a task that the help text supports.

  • Place help for an overarching task at the top of the form.
  • Place help for a particular section of the form below the section heading.
  • Place help for a specific field below the input.
Popovers
Usage

Popovers are a great way to guide users toward a subtle but important element on the page or modal. Users can display popovers when they interact with their trigger elements, and you can also display popovers programmatically when a page loads. It is appropriate to use popover help in cases where:

  • An element is important but easily overlooked. For example, use a popover for a new filter option, a view switcher, or an isolated context menu.
  • An element requires special attention when it loads because user research indicates a problem with discoverability.
Preloaded Help Widget
Usage

Blackbaud developers can preload the Help Widget to display user assistance that is tailored to the first time that users load a page. Use this technique to introduce users to new or novel functionality. Do not use it to walk users through processes.
