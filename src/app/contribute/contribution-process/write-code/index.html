Follow these guidelines before you write code for SKY UX.

  1. Launch a command prompt, cd to the folder where you cloned your branch, and run skyux watch --coverage library.

  2. Launch a second command prompt, cd to the folder where you cloned your branch, and run skyux serve. This launches the visual testing pages where you can view updates based on your code changes. Depending on the nature of your changes, you may need to update the visual regression tests. Any changes will impact the visual e2e tests, and you may need to reset the baselines after you make changes to get the build to pass.

    The SKY UX CLI authors SKY UX demos. If this is your first time using SKY UX Builder, then install the CLI as a global NPM package (npm i -g @skyux-sdk/cli) before you run skyux serve.

  3. When you save code changes, the code compiles, static code analysis is performed, unit tests run, and the page refreshes with the latest changes.

Keep in mind that when you create components, you must add public exports through ./src/app/public/index.ts to allow users to import your component modules. The public exports provide entry points for the bundles that SKY UX deploys.

Also note that the recommended change detection strategy for SKY UX components is to use the OnPush property instead of Default.

For Blackbaud developers, additional style guidelines for authoring SKY UX components are available in our team documentation site. This internal Blackbaud-only site is not available to external developers.
