Contribute to SKY UX We encourage contributions from all SKY UX users. To contribute, follow this step-by-step guide. Keep in mind that before you start, you should install the SKY UX prerequisites. File an issue

Before you start any work, file an issue. If an issue already exists, comment on it and tag the SKY UX team so that we know you intend to work on it. The sooner we discuss potential code changes, the better.

SKY UX provides functionality through multiple repositories that target different areas of development, so use the packages matrix to find the appropriate repo to file your issue.

Discuss the contribution with the SKY UX team

After you file an issue, meet with the SKY UX team to discuss architecture and implementation details, requirements, and timelines. We will assign a developer to your issue to help answer questions.

Keep in mind that changes that impact the UI, such as new components or changes to existing components, require approval from the Blackbaud design council. Contributions that do not have design approval must include detailed design specifications. The design approval process can take significant time if components are complex or designs require significant changes, so account for that in your development timeline. If you need help with the design approval process, ask the SKY UX team.

Write the code for your feature

Create a fork of the appropriate repo, and then write your code and push it to a kebab-cased feature branch. Don't write tests or documentation yet.

To ensure that the SKY UX team can review and accept your contribution in a timely fashion, follow our contribution guidelines.

Request a preliminary review

Create a pull request with the initial implementation, functionality, and styling, and tag the SKY UX team to request a review. Make changes and request additional reviews as necessary. For large changes that effect more than 50 files, please submit changes in several small pull requests instead of one large pull request.

Wait until after the initial review to add tests and documentation because implementation details may change.

Add tests Write tests for your feature. All code requires 100 percent unit test code coverage. Components also require visual regression tests and accessibility tests. Request a final review Tag the SKY UX team to review the complete contribution. We review the implementation, testing, functionality, styling, localization, and accessibility. Document your contribution

Write documentation for your contribution or file an issue in the SKY UX docs repo to work with a SKY UX docs writer. All new components must include demos to accompany the reference documentation in the Components section of the SKY UX website.

Write or update the documentation for your contribution. For components, file an issue in the SKY UX docs repo and work with a SKY UX docs writer. You can find more information on writing docs here.
