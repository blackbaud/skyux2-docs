Usage

A flyout is a container that lays over page content to display large amounts of supplementary information related to user tasks. A flyout can contain lists, record views, graphs, or bespoke information displays.
When to use a flyout

Use a flyout to open records in a list so that users can return to the list view after completing quick tasks on record pages.

Use a flyout to drill into an analytical insight so that users can quickly glance into what makes up that insight without spawning another workflow.

Use a flyout to drill into child records so that users can retain the context of the parent record. The flyout only covers part of the current page, so users can refer back to it for context.
When not to use a flyout

Do not use a flyout to navigate from one record to another instance of the same record type. For example, if users on a constituent record select a link to a different constituent record, navigate to the new record. Do not open it in a flyout.

Do not use a flyout to navigate from a child page to its parent page. For example, if users on a gift record select a link to return to a constituent record, navigate to a full-page view of the constituent record. Do not open it in a flyout.

Do not use a flyout when another flyout is already. If users select a link in the flyout, close the flyout and navigate to the new link location. In other words, limit flyouts to a single drill-in. Any further drilling opens a full page.

Do not use a flyout on pages that don't make sense when it closes. For example, on a page to enter grades for a class of students, displaying the names of students doesn't provide value without the grade entry form, so don't use a flyout for the grade entry form.

Do not use a flyout to display a small amount of additional information. The flyout is for large amounts of supplementary information. To display smaller amounts of information, use a popover component or an expandable repeater.

Do not use a flyout to display items that need a long time to load such as system-generated reports. Instead, let users continue to use the system, and use an alert component to let them know when the content is available.

Do not use a flyout to link to external sites. Open external sites in new tabs. For example, when users select a constituent's address, a Google map of that address open in a new tab.

Do not use a flyout for tasks where users always need the full-screen view of details, such as expanded views of tiles. Navigate to a full page instead.
Interactions
Background page remains active

The background page is active behind the panel. Clicking on a link for another record in the background page will update the panel to load the new record. This is different than clicking on a link inside the flyout which will navigate away from the current page.

Clicking on a non-interactive element on the background page should close the flyout. On touch devices, touching a non-interactive element on the background page should not close the flyout, since the user will need to touch the background page in order to scroll.
Independent scrolling

The panel and the page scroll independently without clicking. If a user touches, hovers, or focuses on the background page when scrolling, only the background content will scroll. Similarly, if a user touches, hovers, or focuses on the flyout when scrolling, only the flyout will scroll. This allows the user to get to the content they want in the panel without losing any context of the page they are on.
Drag to resize

The flyout has a left border which can be clicked and dragged to expand the size of the panel. The panel can be resized anywhere between 320px and the full screen size. When the panel is dragged to the full size of the screen, it will automatically slide back 20px to the right so that a portion of the original page is still in view. The default size of the panel is 50% of the screen (or 320px, whichever is larger). A user can resize the panel, and the system will remember their new preferred size globally.
Help panel

The flyout covers the omnibar, but the help remains visible in the upper right corner of the screen. The help topic should understand that the flyout is open and provide appropriate guidance based on the context.
Modal dialog behavior

When a user takes an action in the flyout that launches a modal dialog, the modal dialog should cover the entire page.
Options
Next/previous row buttons

When the flyout is used in a list to view records, there are buttons in the panel toolbar to go to the next record or the previous record. Clicking on the next button will open the record in the next row that shares a type with the list. For example, in a gifts list, if a user has the flyout open on the constituent record for the first gift in the list, clicking on the next button will open the gift record of the second row in the list. When the next/previous option is turned on, there is a visual indicator on the row in the list to show which row is currently selected.
List toolbar

When viewing a list in the flyout, the toolbar should give users the option to leave the "preview" list and create a new list for further manipulation. Clicking on the create list button will navigate to a new page containing the new list.
Responsiveness
Mobile behavior

The flyout should only be used when the screen is larger than 480px because it covers too much of the screen at smaller resolutions, thus losing much of its value. For smaller screen sizes, the system should navigate directly to the appropriate record or list.
