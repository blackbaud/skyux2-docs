Wizards guide users through pre-defined steps in a particular order to represent a series of related decisions in a userâ€™s direct workflow. Wizards illustrate the step-wise nature of tasks and reinforce progress toward final goals as users complete steps. Wizards are most useful when users must perform steps in a particular order.

To create a wizard, place a progress indicator inside a modal and set its displayMode property to Horizontal.

This wizard implementation replaces a previous version that was deprecated because of limitations that interfered with complex configuration tasks. The deprecated wizard used the tabs module within a modal component and set tabStyle="wizard" on the sky-tabset component to guide users through a set of pre-defined steps. We recommend that you create wizards with the progress indicator component instead of this deprectated implementation. Specifies whether to display the progress indicator vertically or horizontally. For modal wizards, set the display mode to Horizontal. For information about other types of progress indicators, see the progress indicator component.

The progress indicator nav button component allows users to navigate the steps in modal wizards. We recommend against using it in waterfall pages and passive progress indicators.

Specifies the progress indicator component to associate with the nav button. Specifies the label to display on the nav button. Specifies the type of nav button to include. Options include finish, next, and previous. Indicates whether to disable the nav button.

Modal wizards guide users through tasks that require a series of related decisions, but progress indicator components can also create waterfall pages to visually represents progress through a series of sequential steps. For guidance on whether to use modal wizards or waterfall pages, see the progress indicator UX guidelines.

Use modal wizards when:

  • Tasks are best presented as multiple steps that users can complete in a single sitting.
  • Tasks do not result in records or settings with page-level views.
  • The interaction complexity of each step is low. For example, steps use standard form inputs and controls.
  • Tasks include no more than six steps.
