Use when

Use erro messages to communicate system statuses to users when the system encounters unexpected errors.
Do use error messages when system errors occur.

Don't use when

Don’t use error messages to display information about background processes. Use toasts instead.
Don't use error messages for background processes.
Do use toasts to inform users about background processes.

Don’t use error messages when the status of specific content requires user attention, but there is no system error. To notify users of a broken status on a workflow or record, use alerts instead.
Don't use error messages to highlight content that users must address when no error occured.
Do use alerts to call attention to issues that require user interaction.

Don’t use security error messages when users lack permission to view portions of a page or modal such as a tile on a record page. Hide the content from view instead.
Don't use security error messages in portions of a page or modal. Hide the restricted content instead.

Error type

Use the broken error type in situations where the underlying content doesn’t load, or when an issue such as a 500 error reaches the server.

Use the under construction error type when a page is under maintenance or experiences a known outage.

Use the not found error type when a page fails to load because of a 404 error.

Use the security error type when a user doesn’t have access to view content.

Image and action

Include the optional image and action when an error message is the only content on the page. Tiles have limited space and may not be important to users' immediate tasks, so exclude the images and actions and only display the title and description.
Exclude images and actions when error messages only affect portions of the content on a page.

Style and tone

Make error messages brief and use a conversational tone. Explain why the error occured, and when possible, offer a solution to help users address the problem.

Errors frustrate users and indicate serious problems, so error messages must be thoughtful and helpful. Avoid humor, limit the user of "please" and "sorry", and avoid negative constructions such as "you can't" and "you don't".

The error component, including all of its title and description text, is always centered within its container.
