The action button component creates a large button with an icon, header, and details. These buttons are used in sets to present users with multiple, branching options to move forward with a task.

The sky-action-button-container component insures that all inner sky-action-button coponents have consistent height and spacing. The sky-action-button component consists of three inner components: sky-action-button-icon, sky-action-button-header, and sky-action-button-details.

Fires when the user clicks the action button. Specifies an icon to be used from the Font Awesome library. For example, to use the fa-filter icon, use filter as the iconType.
Usage

Use action buttons when:

  • Users initiate tasks, often with a page's primary action.
  • Tasks include 2 to 6 methods of completion.
    • Users must choose exactly one method.
    • If tasks have more than 6 methods of completion, use a different selection input and interaction.
  • Methods of completion are system-defined, not user-defined.
  • Methods of completion mostly determine how tasks proceed.
    • This choice is made at or near the beginning of the task
  • The outcome of end result will be largely the same, regardless of the method

Action buttons must represent a necessary step in any task they inhabit - they should not be used in ways where they can be skipped, avoided, or answered later. Present them in a way that isolates the choice they represent and removes other decisions, different tasks, or other content. Take advantage of the "details" line in the button to give users context or explain the differences between the options.

When to use action buttons

Use action buttons immediately after users initiate tasks to differentiate the methods of completion.

Use action buttons on pages or modals when tasks are underway and the only relevant choice for users is how to complete tasks. Do not display other inputs or choices until users select an action button. Then hide the action buttons or close the modals.

When not to use action buttons

Do not use an action button by itself. Action buttons should not serve as primary calls-to-action on pages where other actions are available to users.

Do not use actions buttons to present different methods of completing tasks before users initate the tasks. Action buttons should not compete against other content for user attention.

Do not use actions buttons when the consequential differences between the choices are a small minority of the scale of the task, or when the choice would not be made at the beginning of the task. Use a different selection input and progressive disclosure to reveal additional content, if needed.

Task placement and behaviors

Use a set of action buttons inside a modal when no page already exists, or the remaining bulk of the task after the action button decision takes place inside a modal. This is common when creating a new record that only requires a standard modal during different versions of its 'Add' process.

  • Use the default size for 2 action buttons, and use a large modal for 3 or more.
    • Center the action buttons inside the modal and wrap to another row if needed.
  • When the user selects a choice, dismiss this modal and open whatever container is appropriate for the remainder of the task.
    • If users cancel or abandon the task, do not return to the modal.
  • Always provide a "Cancel" action inside a modal with action buttons.

Use action buttons directly on pages or tabs that include add or configuration processes but no other tasks. Most add processes use the modal example above.

  • If necessary, include a question above the action buttons to explain the decision being made.
  • Center the question and action buttons inside the page.
  • When users select one of the action buttons, replace them with the next step of the task.
  • If users leave the page without selecting an action button, treat the record or process as "incomplete" and place any appropriate status indicators on its parent view.
    • An explicit "Cancel" or "Delete" action underneath the action buttons and be used if it is valuable for the user to delete the record entirely from this point.

Use action buttons directly inside tabs instead of modals when users add user-defined tabs and select how populate the content.

  • A "Cancel" action under the action buttons is not necessary because users can remove user-defined tabs on the tab control.
