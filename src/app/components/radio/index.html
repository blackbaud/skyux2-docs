The radio button component is a selection input for choosing one option from a small set.

Use when

Use radio buttons when the user must choose one option from a set of 2-5. Use radio buttons in a modal, where the user will confirm the decision before changes take effect.
Do use radio buttons for single selections when the number of choices is low.

Use radio buttons if the user’s decision will make a small number of other inputs relevant or irrelevant. The conditional inputs are visible and disabled when their activating radio is not selected.
Do let radio buttons conditionally enable more inputs as part of a task.

Use an icon radio group when choices can be clearly communicated with icons, and can be combined under a single field or question.

Do use icon radio groups when choices can be clearly illustrated with icons.

Use individual icon radio buttons when the radio group is spread across multiple rows or objects.
Do use icon radio buttons when the choices cannot be grouped together.

Don't use when

Don’t use radio buttons when the set of available choices is more than 5.
Don't use radio buttons when there are many choices.
Do use different selection controls for more than 5 choices.

Don’t use radio buttons when one choice is the exact opposite or absence of the other. Rephrase the question and use checkboxes instead.
Don't represent two exact opposites with radio buttons.
Do use checkboxes for choices between exact opposites.

Don’t use radio buttons when the user must make a choice that will significantly change the rest of a workflow. Use action buttons instead and place that decision at the beginning of the task.
Don't use radio buttons for decisions that will have a drastic effect on the rest of the decisions in a workflow.
Do use action buttons for decisions that branch the user into signficantly different workflows.

Don’t use icon radio buttons if the difference between the choices cannot be clearly communicated with icons. Use traditional radio buttons to provide the needed context with text.
Don't use icon radio buttons when the icons may be unclear.
Do use radio buttons with text labels when icons are not immediately obvious.

Don’t use ungrouped icon radio buttons when the choices could be grouped together on the same line.
Don't use individual icon radio buttons when they could be grouped.
Do use icon radio groups when possible.

Text label

When radio buttons are in grid columns, the text label is not required if you use a column heading. Use text labels for all other states.

Radio type

Icon radio buttons can set a background and border color to match an implied “status” (success, danger, etc.) that corresponds to the types of status indicator.

Radio buttons may or may not use a preceeding question or statement to provide context for the options. When one is not included, provide the context for the consequences of the choice in the radio button text labels themselves.

Icon radio buttons use the FontAwesome 4.7 icon library. See the SKY UX icon dictionary for a list of common icons.

Note: Icon radio buttons must still use a label or labelledby input for accessibility.

Arrange groups of radio buttons in a vertical stack to make it as easy as possible to see all options, scan their labels, and see which option is selected.
Do stack radio buttons vertically.
Don't arrange radio buttons horizontally.

Use standard SKY UX form group spacing to separate radio button groups from the form elements around them.
Do visually separate radio groups from the inputs before and after them.

Guidelines

  • Form design

The sky-radio-group component wraps sky-radio components to organize radio buttons into a group. It is required for SKY UX radio buttons on Angular reactive forms to manage the selected values and keep the forms up-to-date. We recommend using the sky-radio-group component with all radio buttons.

If the radio button group includes a visible label on the screen, use the ariaLabelledBy property instead. This property accepts string values. If the radio button group does not include a visible label on the screen, use the ariaLabel property instead. This property accepts string values. This property supports accessibility by ensuring that focus is placed on the currently selected radio button or on the first or last button when no radio button is selected depending on how users navigate to the radio button group. For a group of radio buttons, use the fieldset and legend elements to provide semantic structure. For people using assistive technology, the fieldset element groups the radio buttons to indicate the number of items in the group, and the legend element provides a description for the group. For more information about how to use fieldset and legend, see WebAIM examples and guidance.

Also for a group of radio buttons, use the tabIndex property to specify an index for all radio buttons in the group. This places focus on the currently selected radio button. If no radio button is selected, it places focus on the first or last button depending on how users navigate to the radio button group.

For icon radio buttons, use the label or labelledby input. For people using assistive technology, the visible icon may not convey sufficient information.
